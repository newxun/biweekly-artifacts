# Protocols: 确认协议、专项子 Agent 与交接文档

> 由 `SKILL.md` 引用。在**要产生副作用**、**考虑派子 Agent**、或**生成/更新交接文档**时读取。

## 确认协议（副作用前置）

任何触发硬边界 #5 的动作前，按风险取一档确认，不能先执行再解释：

- **轻量确认**（只读实验、隔离临时目录内、无持久/外部副作用）：一句话说明「要做什么 + 预期作用 + 无持久副作用」，获得同意即可执行。
- **完整修复批次**（改仓库文件/Git/服务/网络/迁移/装依赖/留进程等）：使用下述模板。

```markdown
## 修复批次确认

- 现象与证据：
- 根因判断及证据等级：
- 不修改的影响：
- 拟修改文件：
- 拟修改内容：
- 拟执行命令：
- 拟修改的非文件状态或外部目标：
- 网络访问的目标与传输内容：
- 是否保留进程、端口或容器：
- 影响范围：
- 项目修复还是临时绕过：
- 清理、恢复和回滚方式：
- 修改后的验证方式：

是否允许执行这个修复批次？
```

获批后：在 Git 仓库中用隔离分支或工作树（无法隔离则停止并说明阻断）；保留用户原有未提交改动；只改确认范围内文件；展示实际 diff；跑约定验证；在报告中记录修改、原因与结果。一次授权不扩展到后续未说明的修改。报告文件同属项目文件，除非用户明确授权本生命周期持续维护某个指定的、规范化后的普通 Markdown 文件（写入前确认它不是符号链接，且授权不扩展到源码或配置）。

## 可选专项子 Agent

默认由主流程完成全部阶段。仅当**同时**满足才启用一个专项子 Agent：问题边界明确且可独立调查；主流程已掌握足以裁剪的上下文；常规调查无法得可靠结论或隔离上下文能显著降混乱；子 Agent 无需改项目文件、请授权或控制共享运行状态。适合的任务是复杂依赖冲突、特定构建失败、独立历史调查；不因仓库大就默认并行派发。

给子 Agent 的输入只含：一个具体问题、已知证据与冲突、允许的范围与工具、固定输出（结论/证据/未确认项/建议下一步）。子 Agent 继承只读范围、秘密保护与网络边界，不得再委派；主流程自行核验其关键结论并负责最终判断。**平台不支持子 Agent 时，全部串行完成，不视为缺陷。**

> 注：本 Skill 本身是**人在环、跨会话**的主线工作流，不应整体被塞进一个非交互的子 Agent；子 Agent 只用于上面这类边界明确、可自治、无需用户确认的专项调查。

## 交接文档与会话交接

交接文档必须自包含，不依赖旧对话。每份记录仓库、分支、commit SHA、工作树状态、环境标识、生成时间；**基线用命令取值（`date -u`、`git rev-parse HEAD`、`git status`），不凭记忆**。新会话接手时先比较基线：一致则复用；代码/配置/依赖/环境已变化时只重验受影响结论（用 `git diff <旧SHA>..HEAD --stat` 定位命中路径，对应结论降为 `[代码证据]` 或 `[未确认]`）；无法判断影响范围时把相关结论降级并说明需重验什么。

输出时机：

- Phase 2 结束且尚无任务 → 项目与环境就绪检查点。
- Phase 3 结束 → 最终开发交接文档。
- 用户表示即将结束或切换会话 → 立即出当前进度检查点。

默认使用同目录的 `development-readiness-report-template.md`（章节裁剪见模板顶部说明），写入目标仓库前按确认协议；用户已授权指定报告文件时可持续更新，无需对同一授权范围重复询问。文档须让新的 Coding Agent 只读报告和任务描述即可继续，至少含：就绪结论与立即行动清单、项目与业务速览、已验证环境与命令与服务、任务相关代码入口/边界/风险、开发测试调试指南、已解决与未解决阻断、证据冲突与未知项与需咨询角色、证据记录与实际命令。

## 最终就绪输出

完成 Phase 3 后输出：

```markdown
## 开发交接已准备

- 环境状态：就绪 / 有限就绪 / 未就绪
- 任务状态：就绪 / 未就绪 / 未评估（尚无任务）
- 交接完整度：最终交接 / 检查点
- 当前任务：
- 推荐修改范围：
- 环境验证：
- 开发与调试方式：
- 最大风险：
- 未解决阻断：
- 交接文档：

下一步：
- 任务状态为 `就绪`：在新的 Coding Agent 会话中读取交接文档并开始实现。
- 任务状态为 `未就绪`：先解除列出的阻断，不得开始实现。
- 任务状态为 `未评估`：补充具体任务后继续 Phase 3。

任何情况下都不要由本 Agent 继续编码。
```

若环境或任务状态不是 `就绪`，不得用「准备完成」掩盖阻断：标题改为「开发交接检查点」，明确用户必须先处理什么；环境为 `有限就绪` 时还要记录用户是否明确接受允许范围与残余风险。
