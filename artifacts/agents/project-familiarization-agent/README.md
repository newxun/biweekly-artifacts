# Project Familiarization Agent

## Problem

研发团队中，开发者可能被临时借调到陌生项目组开发需求。真正开始编码前，他们往往需要先理解项目、业务、负责人、代码规范、历史坑、隐含约定和风险边界。

这些信息分散在 README、代码、提交历史、PR、Issue、测试、配置和团队成员经验中，手动搜寻成本高，也容易遗漏高风险信息。

## Goal

帮助开发者在编码前快速完成项目熟悉，明确：

- 已经了解什么。
- 还缺少什么。
- 应该优先阅读什么。
- 哪些区域风险较高。
- 哪些问题需要提前向项目成员确认。
- 是否已经具备安全开始开发的条件。

Agent 的职责到开发开始之前结束。

## Use Cases

- 临时借调到陌生项目组。
- 第一次参与某个仓库的需求开发。
- 接手一个已有模块但缺少业务上下文。
- 开发前需要快速建立风险地图。

## Not For

- 自动开发。
- 自动修改业务代码。
- 自动做架构设计。
- 自动做 Code Review。
- 自动完成需求拆解和排期。

## Usage

将 [`agent.md`](agent.md) 的内容复制到任意支持自定义 Agent / System Instructions 的编程工具中，例如 Cursor、Claude Code、Copilot 或 Gemini CLI。

建议启动语：

```text
我被临时借调到这个项目，需要在开始编码前熟悉项目。当前需求是：<描述需求>。请引导我完成项目熟悉。
```

如果当前需求还不清楚，可以这样启动：

```text
我刚加入这个项目，还没有完整需求。请先帮我建立项目认知，并告诉我开始接需求前应该确认哪些信息。
```

## Inputs

Agent 可以使用：

- 当前仓库文件。
- README、docs、配置文件、测试和脚本。
- Git 历史、提交信息、分支约定。
- 用户提供的需求、Issue、PR、任务描述。
- 用户提供的团队成员、负责人或业务背景。

## Outputs

Agent 持续维护 Checklist，而不是一次性输出长报告：

- 已了解
- 建议优先了解
- 建议阅读
- 高风险区域
- 建议咨询项目成员
- 开发前仍需确认

## Human Confirmation

必须由开发者确认：

- 当前需求目标。
- 修改边界。
- 业务规则。
- 高风险模块是否允许修改。
- 需要咨询的负责人。
- 是否可以开始编码。

## Limitations

- Agent 不能知道仓库之外的团队历史和口头约定。
- Agent 不应猜测负责人或业务决策。
- Agent 不负责写业务代码。
- Agent 不应把仓库分析包装成确定结论，除非有明确证据。

## Roadmap

- 增加不同项目类型的启动问题集。
- 增加项目风险地图模板。
- 增加“开发前准备完成度”轻量评分。
- 沉淀真实借调场景中的常见问题。
