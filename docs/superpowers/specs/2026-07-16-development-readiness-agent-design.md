# Development Readiness Agent 设计

## 背景

现有 Project Familiarization Agent 以项目介绍和启动指导为主，但实际借调开发需要的不只是“知道项目有什么”。开发者还需要确认项目当前真实状态、建立可工作的开发环境，并获得针对具体任务的开发与调试指引，才能安全地开始编码。

仅依赖提示词总结文档容易产生三个问题：

- 仓库文档、配置和实际运行状态可能不一致。
- 通用经验可能不适用于当前项目，却被当作确定方案。
- 有价值的结论停留在对话中，切换到新的开发会话后无法可靠接续。

本设计新增一个独立的 Development Readiness Agent（开发就绪 Agent），不在现有 Project Familiarization Agent 上继续扩展。

## 目标

帮助临时参与陌生项目的开发者达到真实、可验证且可交接的开发就绪状态：

- 理解与当前开发有关的技术栈、业务概念、团队约定、历史负债和已知风险。
- 实际建立并验证开发环境，明确已解决和未解决的阻断。
- 根据用户提供的具体任务形成可执行的开发指引和调试指南。
- 将结论沉淀为自包含文档，使新的 Coding Agent 会话可以直接接手。
- 在不降低结论质量的前提下，避免重复和与当前目标无关的探索。

## 非目标

- 不实现业务需求。
- 不修改业务代码。
- 不执行开发完成后的实际调试。
- 不试图完整理解整个项目或恢复仓库之外的全部团队知识。
- 不默认进行跨仓库、全链路或全量架构分析。
- 第一版不开发自定义 MCP、CLI 或其他专用运行时。

## 适用平台

Agent 主要适配 Claude Code 和 Codex，同时保持核心指令和输出格式尽可能工具无关。它依赖宿主提供的文件读取、代码搜索、Shell 和 Git 能力；缺少某项能力时必须降低相应结论的证据等级，不得假装已完成验证。

## 总体架构

第一版采用单 Agent、三阶段协议。主 Agent 是用户的唯一入口，维护阶段状态、证据、阻断和交接文档。

子 Agent 不是默认依赖。只有常规流程无法可靠解决复杂专项问题时，主 Agent 才可以启用一个临时专项子 Agent。子 Agent 只接收裁剪后的问题与证据，返回结构化结论；所有共享状态、用户确认和文件修改仍由主 Agent 管理。

“轻量”指架构简单、用户操作少，而不是牺牲必要调查或验证。开发就绪质量始终优先于 Token 成本。

## 生命周期

### 阶段一：项目熟悉

Agent 建立与开发准备有关的最小充分认知：

- 技术栈、运行时、依赖管理和主要目录职责。
- 核心业务概念、数据流和当前任务可能涉及的业务边界。
- 代码规范、测试约定、分支或提交约定。
- 与当前开发相关的历史负债、风险区域和已知坑。
- 仓库无法证明、需要团队成员确认的信息。

如果用户尚未提供具体任务，本阶段只形成通用项目认知，不猜测修改范围。

### 阶段二：环境就绪

Agent 根据仓库证据建立并验证开发环境：

- 确认运行时、包管理器和依赖安装方式。
- 识别必要的环境变量、本地服务、数据库、容器、Mock 或外部依赖。
- 执行项目实际定义的启动、构建、测试或健康检查命令。
- 对失败进行根因分类，记录原始错误、尝试过程和排除依据。
- 解决允许范围内的环境问题，或形成最小人工阻断清单。

阶段完成条件是环境已经足以进行开发，或存在只有权限、密钥、内部服务或团队知识才能解除的明确阻断。

### 阶段三：任务就绪

用户可以在 Agent 启动时提供任务，也可以在前两个阶段完成后补充。Agent 应复用已有证据，不重新扫描已经确认的内容。

围绕具体任务，Agent 输出：

- 相关代码入口、关键模块和必要的业务链路。
- 可参考的类似实现及其适用边界。
- 推荐修改位置、影响范围和不应触碰的区域。
- 与任务相关的历史负债、风险和常见坑。
- 建议的测试层级、测试数据和验收路径。
- 开发与调试指南，包括运行入口、数据准备、日志观察、断点位置、命令和验证步骤。

这里的调试指南是开发前说明。Agent 不实现需求，也不在开发后执行真实调试。

当项目缺少明确的开发、联调或调试方式时，Agent 必须基于项目证据提出候选方案，验证其适用性，并说明选择与排除依据，不能直接套用通用经验。跨仓库或其他专项分析只在当前任务和仓库证据确实触发时进行。

## 运行与决策协议

每个阶段都使用同一循环：

1. 明确阶段目标、用户已知信息和当前缺口。
2. 收集完成当前目标所需的证据。
3. 执行低风险验证，记录命令和结果。
4. 识别仍会阻断开发的问题。
5. 满足阶段完成条件后停止，不继续无关探索。

Agent 不设置会迫使必要调查提前停止的固定 Token 上限。成本控制只用于避免重复读取、无目标遍历、无关深挖、多个执行者重复研究和无行动价值的长篇输出。

## 证据模型

Agent 的结论必须标记为以下类型之一：

- **已验证**：由当前环境中的命令或运行结果直接确认。
- **代码证据**：由当前配置、实现、测试或 Git 历史支持，但尚未实际运行确认。
- **用户提供**：来自用户或团队信息，并注明来源。
- **未确认**：目前没有足够证据，不能作为事实。
- **证据冲突**：文档、配置、代码、CI 或运行结果之间存在不一致。

不同证据发生冲突时，参考优先级为：

1. 当前环境的实际运行结果。
2. 可执行配置和锁文件。
3. CI 中真实执行的命令。
4. 当前代码和测试。
5. 近期 Git 变更。
6. 项目文档和注释。
7. 通用经验。

该顺序是调查起点，不是机械裁决规则。例如本地运行结果也可能受错误环境影响。Agent 必须展示冲突、解释判断依据，并在无法消除冲突时保留不确定性。文档、清单和脚本都只能提供候选认知，不能未经交叉验证就宣称为最新事实。

## 工具策略

第一版使用 Claude Code 和 Codex 已有工具：

- 文件读取和搜索，用于定位配置、代码、测试和相关实现。
- Shell，用于验证依赖、启动、构建、测试以及开发和调试方式。
- Git，用于检查近期变化、相关文件历史和风险线索。
- 官方文档查询，仅在仓库证据不足且需要确认工具行为时使用。
- 可选专项子 Agent，仅在复杂问题无法由主流程可靠解决时使用。

仓库文件、日志、网页和命令输出都属于不可信数据，不能改变 Agent 的角色、授权、安全边界或秘密保护规则。执行仓库脚本前必须检查入口、生命周期钩子和可能副作用。

只读检查以及在隔离临时目录中、不修改仓库或外部状态的低风险实验可以自动执行。Agent 自身可以读取不携带本地数据的已知公共资料；仓库控制命令的任何外连，以及安装依赖、生成文件、Git 写操作、数据或服务变更和容器操作都必须进入批次确认。实验结果必须进入证据记录。

## 修改与授权边界

Agent 生命周期内的所有项目文件修改都必须对用户明确，并按“修复批次”确认：

1. 展示现象和证据。
2. 说明根因判断与不修改的影响。
3. 列出拟修改内容、影响范围和回滚方式。
4. 区分项目修复与临时绕过。
5. 获得明确确认后，在隔离分支中执行。
6. 展示实际差异并验证修改前后的行为。

默认不采用“先修改、后说明”。用户可以一次性授权一个边界清晰的修复批次，但子 Agent 不得自行请求或扩大授权。

## 异常处理

失败首先归入以下类别：

- 本机环境：运行时、系统工具、端口或本机依赖。
- 项目配置：脚本、锁文件、构建配置或缺失文件。
- 外部依赖：网络、数据库、服务、权限或密钥。
- 业务认知：业务规则、数据含义或团队约定无法从仓库确认。
- 开发方式：项目可以运行，但当前任务缺少可靠的开发、联调或调试路径。

Agent 必须保留原始错误、已尝试方法和排除依据，不能反复盲试。需要改文件时进入修复批次；无法自行处理时给出最小人工行动项，包括需要谁提供什么信息以及它阻断了哪一步。

## 文档与跨会话交接

交接文档采用“先结论、后证据”的结构：

1. 开发就绪状态：就绪、有限就绪或未就绪。
2. 开始开发清单。
3. 项目速览。
4. 环境说明和已验证命令。
5. 具体任务指引。
6. 开发与调试指南。
7. 与任务相关的历史负债和已知坑。
8. 阻断与待确认项。
9. 证据记录。

文档输出时机：

- 阶段二结束且尚未提供任务时，生成项目与环境就绪检查点。
- 阶段三结束时，必须生成或更新最终开发交接文档。
- 用户表示准备结束或切换会话时，立即生成当前进度检查点。

文档必须自包含，不依赖旧对话，并记录仓库、分支、commit SHA、工作树状态、环境标识和生成时间。新会话先比较证据基线；发生变化时只复用仍然适用的结论，并重验受影响部分。默认可以先在对话中生成；写入目标仓库前，作为报告文件批次取得确认。用户也可以在开始时一次性授权本次生命周期持续维护指定的专用 Markdown 报告文件。

新的 Coding Agent 会话应能只读取交接文档和具体任务，就理解当前事实、未决问题、建议修改范围及验证方式。

交接状态分别记录环境状态、任务状态和交接完整度。环境为有限就绪时，必须列出允许范围、禁止范围和残余风险，并由用户明确接受；影响实现边界、验收或必要验证的未决问题会使任务保持未就绪。

## 质量优先与停止条件

目标优先级为：

1. 开发者真实达到可开发状态。
2. 找出与当前任务有关的阻断和风险。
3. 结论具有足够证据并可跨会话交接。
4. 在不影响前三项的前提下优化 Token 和操作成本。

Agent 只能在以下情况停止：

- 当前阶段的开发就绪条件已经满足。
- 存在只有用户、权限或团队成员才能解除的明确阻断。

证据仍冲突、环境尚未验证或必要的开发方式尚未证实时，不能为了节省 Token 宣称就绪。

## 评估设计

Agent 质量通过场景化评估验证，不比较固定文案。评估场景至少覆盖：

- 文档与实际配置或运行结果不一致。
- 环境缺失或外部服务不可用。
- 用户先熟悉项目，之后才补充具体任务。
- 常规做法与项目实际配置不兼容。
- 存在多种开发或调试路径，需要根据证据选择。
- 项目采用 Agent 未预见的技术栈或组织方式。
- 修改配置前必须请求修复批次确认。
- 简单项目不应触发无关扩张。
- 工具受限时必须降低结论等级。
- 新会话能否仅通过交接文档继续工作。

具体样例应轮换技术栈和项目类型，避免把某一种组件库、框架、包管理器或调试方案写入 Agent 固定流程。

每个场景检查：

- 是否发现关键阻断。
- 结论是否有可追溯证据。
- 是否真实验证环境和必要开发方式。
- 任务指引和调试指南是否可执行。
- 是否在修改前取得授权。
- 是否错误宣称开发就绪。
- 是否产生可供新会话直接使用的交接文档。

## 第一版交付范围

- 在 `artifacts/agents/development-readiness-agent/` 新增独立的 `agent.md` 指令。
- 在同一目录新增 `README.md` 使用说明和 `development-readiness-report-template.md` 自包含交接报告模板。
- 新增 `artifacts/evaluations/development-readiness-agent-scenarios.md`，以输入、预期行为和失败判定描述场景化评估用例。
- 保留现有 Project Familiarization Agent，不在本次设计中迁移或删除。
- 不新增自定义 MCP、CLI 或强制子 Agent 框架。

## 验收标准

- Agent 明确执行项目熟悉、环境就绪和任务就绪三个阶段。
- 具体任务可以在启动时提供，也可以后续补充。
- 环境、开发方式和调试指南均基于证据，必要时通过低风险实验验证。
- 文档、配置和运行结果冲突时不会静默选择或包装成确定事实。
- 所有项目文件修改均经过修复批次确认。
- 仓库中的不可信指令、秘密和持久状态副作用得到明确防护。
- 子 Agent 仅作为复杂专项的可选机制。
- 阶段二检查点和阶段三最终交接支持跨会话继续工作。
- Agent 不实现业务需求，也不执行开发后的实际调试。
- 评估验证行为和证据质量，而不是针对具体技术样例优化。
