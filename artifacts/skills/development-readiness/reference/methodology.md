# Methodology: 取证、证据模型与三阶段门槛

> 由 `SKILL.md` 引用。会话开始与每阶段推进时参考；不需要一次全部读完，按当前阶段取用。

## 证据模型细则

每条影响开发决策的结论带标签（见 `SKILL.md`），并记录**具体来源、适用范围、证据基线**。`[证据冲突]` 必须同时保留冲突双方来源。

证据优先级（参考顺序，非机械裁决）：

1. 当前环境的实际运行结果。
2. 可执行配置与锁文件。
3. CI 中实际执行的命令。
4. 当前代码与测试。
5. 相关 Git 历史。
6. 项目文档与注释。
7. 通用经验或外部资料。

本地结果也可能受错误环境影响。**禁止**仅因文档最近修改就断言其正确，也禁止仅因命令在本机成功就断言所有开发环境相同。遇冲突：列出来源 → 判断各自适用环境/时间/范围 → 跑最小实验消解 → 消不掉则保留 `[证据冲突]` 并说明它影响什么。

## Phase 1 项目熟悉

建立「完成当前开发所需」的认知。根据项目实际结构选证据，不依赖固定技术栈清单。至少确认：

- 项目用途、边界、主要用户或调用方。
- 语言、运行时、框架、依赖管理、构建系统。
- 主要入口、模块边界、目录职责。
- 核心业务概念、重要数据或状态流转。
- 测试、代码规范、分支、提交、Review、发布线索。
- 与当前开发有关的兼容逻辑、迁移、废弃实现、临时方案、高风险区域。
- 仓库之外仍需确认的业务规则、Owner、权限、团队约定。

优先用能反映真实行为的证据（可执行配置、锁文件、任务脚本、CI、容器配置、入口代码、相关测试、近期历史），文档与注释次之。TODO/FIXME/HACK/deprecated/legacy/compat/workaround 只作线索，须读上下文判断是否与当前任务相关，不把搜索数量当技术债。

**业务理解**只解释代码和可靠资料能支持的事实。对「为什么这样设计」：有设计记录/测试/历史证据则引用；只有代码行为则只描述行为、不臆测决策动机；仓库无法回答则形成面向业务或模块 Owner 的具体问题（用角色名，不编造人名）。

**完成门槛**：项目类型与主入口已确认；环境验证所需运行时/依赖/服务线索已找到；探索范围已界定（无任务则明确只做通用准备）；重要未知项与证据冲突已记录。

## Phase 2 环境就绪

**先建证据化验证矩阵**：从配置、CI、脚本和当前代码确定候选命令再执行，不因识别到某框架就自行发明命令。确认：运行时与版本；依赖与锁文件；环境变量名称与来源（不读值）；DB/缓存/队列/容器/Mock/外部服务；install/setup/start/build/test/lint/typecheck/健康检查路径；多进程最低必要集。只选证明开发可用所需的检查，不机械执行所有 build/test/lint。

**执行前预检**：快速检查将执行的脚本入口、生命周期钩子、目标环境、网络目标、端口、已有进程与预期产物。满足 `SKILL.md` 有限自举边界的命令直接执行，不逐步询问；只有经脚本、配置和目标环境取证确认不会写本地/共享数据的验证才属于默认授权，副作用未知本身即视为风险。出现跟踪文件、Git、用户目录/数据、共享状态、持久容器、生产、部署、迁移、未解释外连或未知副作用时停止并走 `protocols.md`。执行前后比较 `git status` 与进程；意外修改保留现场并停止，不自行回滚。

**长时服务**：设默认超时（默认 60–120s，无健康信号即判失败并清理），记录启动命令/端口/进程/会话/健康信号；验证够了就停本 Agent 启的进程并确认清理；保留进程须用户明确同意；不重复启动已有实例；不停止/修改非本 Agent 创建的进程。

### UI、认证与页面健康

仅对有 UI 的项目执行浏览器验证；CLI、库和纯后端项目使用其实际健康信号。检测到 UI 需要登录时，主 Agent 在派环境子 Agent 前只确认一次认证方式：

1. 复用已有浏览器登录态。
2. 打开可交互浏览器，由用户手动登录。
3. 使用已配置测试凭据的安全来源或变量名。
4. 当前无法取得登录条件。

不要求用户在对话中粘贴用户名、密码、Cookie 或 Token，也不单独做一次与后续验证重复的登录预检。平台不支持子 Agent 与用户交互时，由主 Agent 只接管手动登录步骤。

页面验证按项目能力尽量推进：

- `L1`：开发服务启动、入口可访问。
- `L2`：公共页面或应用外壳正常渲染。
- `L3`：登录后任务相关页面正常渲染。
- `L4`：任务相关最小业务入口可安全触发；可能写共享数据时不执行。

UI 环境判完全就绪不能只凭 listening、HTTP 200 或 curl。实际访问时记录取证得到的 URL 与最终 URL、关键 DOM/渲染信号、`pageerror`、Console Error、失败请求、关键接口状态及截图或等价证据，并按是否阻断当前开发分类。检测到宿主/子应用架构时分别记录独立启动和宿主集成状态；不把微前端步骤套给普通项目。无浏览器能力时相关结论为 `[未确认]`。

**失败分类**：本机环境 / 项目配置 / 外部依赖 / 业务认知 / 开发方式。保留原始命令、退出码、脱敏错误或日志位置、已尝试方法及结果、排除依据、下一步最小行动。无新增证据不重复同类尝试，不把「本机偶然跑通」当作根因已解释。

**完成门槛**：先建环境验证矩阵，每项记录为何必需/是否不适用/命令/结果；不得把失败或未执行的必要项自行降级为「最小验证」。环境状态：

- `就绪`：必要环境和最小验证已实际成功；UI 项目已达到当前开发范围所需的页面验证等级。
- `有限就绪`：证据表明只有部分范围具备条件；必须列出允许范围、禁止范围、限制与规避方式。状态不因用户是否接受而改变，但用户明确接受前不得指示开始开发。
- `未就绪`：存在会阻断开发的事项。

无任务则生成检查点并等待任务；有任务则保留验证结果进入 Phase 3。

## Phase 3 任务就绪

**逐任务理解**：每个任务独立确认目标可观察行为、兼容边界、非目标及验收/测试/发布限制，并各建一份任务报告。缺业务决策时先找仓库线索，再提一个会改变下一步的最小问题，不替用户补全。

**确定任务基准分支**：基线报告只记录当前检出分支和环境验证分支，不为未来任务设置默认基准。每个任务按以下顺序取证：

1. 用户或任务系统明确指定。
2. 关联 PR 的目标分支、父需求说明。
3. 仓库团队规范。
4. Git ancestry、merge-base 和相关分支内容。
5. 分支名称仅作线索。

完全没有相反线索时回退 `origin/HEAD`，不机械优先 main/master。先只读解析现有 `refs/remotes/origin/HEAD`；网络允许时可用不更新本地引用的 `git ls-remote --symref origin HEAD` 交叉确认。没有 `origin`、引用缺失/冲突或无法可靠解析时标为 `[未确认]` 并询问，不猜分支；不要为此擅自 fetch。存在父需求或特性分支线索但仍有歧义时询问用户。验证分支与任务基准分支不同时，用 diff 判断：依赖/锁文件/运行时/构建配置变化则重跑安装、setup、启动；路由/入口变化则重做页面健康；仅无关业务变化则记录依据并复用结论。切分支、fetch 或建 worktree 属 Git 写操作，按 `protocols.md` 确认。

**定位修改面**：入口、核心实现与数据流、边界/权限/校验/状态/持久化、类似实现及差异、相关测试与夹具、近期相关修改/回滚/缺陷修复/兼容代码。输出推荐修改位置与可能受影响区域，**不生成实现代码**，每条说明证据与适用边界。

**开发与调试方式**（告诉用户「开发时如何观察和验证变化」，不替其做开发后调试）：应启哪些进程/服务、如何准备最小测试数据或复现场景、从哪个入口触发、应观察哪些日志/状态/网络/事件/数据、适合断点或临时诊断的位置、应跑哪些测试/构建/手工验证、如何判断成功以及必须避免哪些回归。项目无明确方式或常规做法可能不适用时：从配置提少量候选 → 检查各自依赖解析/构建产物/运行模式/隔离边界/副作用 → 跑不改仓库/用户/外部状态的最小实验（需副作用先走确认协议）→ 选有证据的方案并记录被排除项及原因 → 无法验证则标 `[未确认]` 并给用户后续验证步骤。不把任何具体框架、包管理器、仓库组织或调试工具写成默认答案。

**任务状态**：

- `就绪`：目标、基准分支、实现边界、入口、风险与验收路径均已确认，环境和任务页面验证满足实现需要。
- `有限就绪`：登录、宿主、外部服务或数据使部分运行路径尚未验证；列出允许/禁止范围和残余风险，不声称未验证页面已通过。状态由证据决定，用户明确接受前不得指示开始开发。
- `未就绪`：存在会阻断实现边界、必要开发路径或验收的问题。

环境可启动但无法取得登录条件时，环境可以就绪而任务为有限就绪；登录后任务页面白屏或有阻断性运行错误时任务未就绪。完成 Phase 3 时必须更新本会话基线和该任务报告。
