# Methodology: 取证、证据模型与三阶段门槛

> 由 `SKILL.md` 引用。会话开始与每阶段推进时参考；不需要一次全部读完，按当前阶段取用。

## 证据模型细则

每条影响开发决策的结论带标签（见 `SKILL.md`），并记录**具体来源、适用范围、证据基线**。`[证据冲突]` 必须同时保留冲突双方来源。

证据优先级（参考顺序，非机械裁决）：

1. 当前环境的实际运行结果。
2. 可执行配置与锁文件。
3. CI 中实际执行的命令。
4. 当前代码与测试。
5. 相关 Git 历史。
6. 项目文档与注释。
7. 通用经验或外部资料。

本地结果也可能受错误环境影响。**禁止**仅因文档最近修改就断言其正确，也禁止仅因命令在本机成功就断言所有开发环境相同。遇冲突：列出来源 → 判断各自适用环境/时间/范围 → 跑最小实验消解 → 消不掉则保留 `[证据冲突]` 并说明它影响什么。

## Phase 1 项目熟悉

建立「完成当前开发所需」的认知。根据项目实际结构选证据，不依赖固定技术栈清单。至少确认：

- 项目用途、边界、主要用户或调用方。
- 语言、运行时、框架、依赖管理、构建系统。
- 主要入口、模块边界、目录职责。
- 核心业务概念、重要数据或状态流转。
- 测试、代码规范、分支、提交、Review、发布线索。
- 与当前开发有关的兼容逻辑、迁移、废弃实现、临时方案、高风险区域。
- 仓库之外仍需确认的业务规则、Owner、权限、团队约定。

优先用能反映真实行为的证据（可执行配置、锁文件、任务脚本、CI、容器配置、入口代码、相关测试、近期历史），文档与注释次之。TODO/FIXME/HACK/deprecated/legacy/compat/workaround 只作线索，须读上下文判断是否与当前任务相关，不把搜索数量当技术债。

**业务理解**只解释代码和可靠资料能支持的事实。对「为什么这样设计」：有设计记录/测试/历史证据则引用；只有代码行为则只描述行为、不臆测决策动机；仓库无法回答则形成面向业务或模块 Owner 的具体问题（用角色名，不编造人名）。

**完成门槛**：项目类型与主入口已确认；环境验证所需运行时/依赖/服务线索已找到；探索范围已界定（无任务则明确只做通用准备）；重要未知项与证据冲突已记录。

## Phase 2 环境就绪

**先建证据化运行计划**：从配置、CI、脚本、当前代码确定候选命令**再执行**；不因识别到某框架就自行发明命令。确认：必需运行时与版本；依赖安装方式与锁文件策略；环境变量名称与来源（不读值）；DB/缓存/队列/容器/本地服务/Mock/外部服务；install/start/build/test/lint/typecheck/健康检查路径；多应用或多进程项目的启动顺序与最低必要集；对微前端子应用，还要确认是否需要宿主应用才能正常渲染。

**安全执行**：**环境自举**（用项目声明的命令装依赖、跑 setup/生成 gitignored 产物、起 dev/build/test/lint、打开运行中的应用做健康检查、用完清理进程）默认授权、无需逐次确认——它是本阶段的核心目的且在满足「自举安全前提」时是安全的（见 `protocols.md`）。执行自举前仍要**快速检查安装/启动脚本的生命周期钩子**，确认没有项目未声明的外连、写用户目录、读凭据等行为（若有则暂停并按 `protocols.md` 处理，见 Scenario 10）。凡**超出自举**、可能改被跟踪文件或产生持久/外部副作用者（如为修环境去改 `package.json`/锁文件/`.npmrc`、起容器、改共享服务），先走确认协议，不先执行再解释。

**应用健康检查（UI 应用必做）**：对有界面的应用（Web / 微前端等），仅「进程 listening / HTTP 200 / curl 通」**不足以**判定环境就绪；必须**实际打开运行中的应用**（可用浏览器工具或 headless 浏览器；无浏览器能力时降级为 `[未确认]` 并给出用户自查步骤），确认关键界面能渲染、控制台无阻断性运行时报错。区分「阻断性报错」（白屏、崩溃、核心功能不可用）与「非阻断提示」（如子应用未挂载宿主时的可关闭弹窗）——前者使环境 `未就绪` 或 `有限就绪`，后者记录为已知提示。此步骤边界清晰、可自治，适合在支持时交由专项子 Agent 执行（见 `protocols.md`）。

**长时服务**：设默认超时（默认 60–120s，无健康信号即判失败并清理），记录启动命令/端口/进程/会话/健康信号；验证够了就停本 Agent 启的进程并确认清理；保留进程须用户明确同意；不重复启动已有实例；不停止/修改非本 Agent 创建的进程。

**失败分类**：本机环境 / 项目配置 / 外部依赖 / 业务认知 / 开发方式。保留原始命令、退出码、脱敏错误或日志位置、已尝试方法及结果、排除依据、下一步最小行动。无新增证据不重复同类尝试，不把「本机偶然跑通」当作根因已解释。

**完成门槛**：先建环境验证矩阵，每项记录为何必需/是否不适用/命令/结果，并**记录验证所在分支**（若与任务基准分支不同，相应结论需在基准分支重验）；不得把失败或未执行的必要项自行降级为「最小验证」。环境状态：

- `就绪`：必要环境和最小验证已实际成功；对 UI 应用，还须已实际打开应用确认可访问、无阻断性运行时报错。
- `有限就绪`：只有部分范围可安全开始；必须列出允许范围、禁止范围、限制与规避方式，且须**用户明确接受残余风险**，否则任务保持 `未就绪`。
- `未就绪`：存在会阻断开发的事项。

无任务则生成检查点并等待任务；有任务则保留验证结果进入 Phase 3。

## Phase 3 任务就绪

**理解任务**：目标可观察行为、必须保持兼容的行为、明确不在范围内的内容、已知验收/测试/发布限制。缺业务决策时先从仓库找可验证线索，再提最小确认问题，不替用户补全。**多任务**时每个任务独立成篇（各建一份任务就绪文件，见 `protocols.md`），不混在一起。

**确定基准分支**：不假设任务基于 main/master。从任务描述/需求文档、相关分支（`git branch -a`、命名约定、父需求分支）、团队约定取证，确认本任务应从哪个分支切出；若前面的环境验证是在别的分支（如 master）上做的，而基准分支与之在依赖/锁文件/相关文件上有差异（用 `git diff <基准分支>..<验证分支> --stat` 判断），则相应环境结论需在基准分支上重验，并在文档中记录。基准分支未取证前不把任务判为 `就绪`。

**定位修改面**：入口、核心实现与数据流、边界/权限/校验/状态/持久化、类似实现及差异、相关测试与夹具、近期相关修改/回滚/缺陷修复/兼容代码。输出推荐修改位置与可能受影响区域，**不生成实现代码**，每条说明证据与适用边界。

**开发与调试方式**（告诉用户「开发时如何观察和验证变化」，不替其做开发后调试）：应启哪些进程/服务、如何准备最小测试数据或复现场景、从哪个入口触发、应观察哪些日志/状态/网络/事件/数据、适合断点或临时诊断的位置、应跑哪些测试/构建/手工验证、如何判断成功以及必须避免哪些回归。项目无明确方式或常规做法可能不适用时：从配置提少量候选 → 检查各自依赖解析/构建产物/运行模式/隔离边界/副作用 → 跑不改仓库/用户/外部状态的最小实验（需副作用先走确认协议）→ 选有证据的方案并记录被排除项及原因 → 无法验证则标 `[未确认]` 并给用户后续验证步骤。不把任何具体框架、包管理器、仓库组织或调试工具写成默认答案。

**完成门槛（判 `就绪` 需全部满足）**：任务目标/实现边界/验收方式已确认；**基准分支已取证确认**且环境在基准分支上仍满足本任务需要；相关代码入口、必要业务链路、类似实现已定位；高风险区域与任务相关历史坑已记录；建议测试与调试路径可执行且有项目证据支持；最终交接文档（基线文档 + 本任务文件）已生成或更新。任何影响实现边界、验收或必要验证的未决业务问题存在时，任务状态必须为 `未就绪`。
