# Project Familiarization Agent 可靠性增强设计

## 背景

当前 Project Familiarization Agent 已经覆盖项目分析、本地启动检查、轻量报告和证据标记。但新的使用反馈进一步指出：

- Agent 与用户讨论后形成的共识、取舍和待确认事项，容易停留在对话里，没有稳定沉淀到文档中。
- Agent 可以帮助识别启动命令，但还需要更明确地帮助开发者掌握开发后的调试方法。
- Agent 需要更严格地区分真实依赖、文档描述、隐含运行时依赖和未经验证的假设。
- Agent 不能被 README、目录名、示例文件、历史残留或表面报错误导，从而给用户错误结论。

这次改进仍然只强化现有 `project-familiarization-agent`，不新增通用 Agent，也不把它扩展成 Coding Agent。

## 设计目标

- 让项目熟悉过程中的关键讨论结论可以被保存、复用和交接。
- 让 Agent 在开发前帮助用户建立可执行的调试路径，而不是只告诉用户项目怎么启动。
- 让依赖判断必须基于配置、代码、脚本或命令输出等证据。
- 让 Agent 对表象信息保持怀疑，明确暴露证据冲突、未知信息和需要人工确认的问题。
- 保持输出轻量，继续服务于“开发者可以安全开始开发”这一边界。

## 范围

本次后续实现应更新以下文件：

- `artifacts/agents/project-familiarization-agent/agent.md`
- `artifacts/agents/project-familiarization-agent/README.md`
- `artifacts/agents/project-familiarization-agent/project-familiarization-report-template.md`

不新增可运行自动化，不改核心目录结构，不新增通用 Agent。

## 行为设计

### 1. 讨论结论沉淀

Agent 应在项目熟悉过程中主动识别“值得沉淀”的信息，而不是只在最后生成一次总结。

应沉淀的信息包括：

- 用户已确认的需求目标、边界和非目标。
- 与用户讨论后确定的方案取舍。
- 开发前必须遵守的业务规则、兼容要求和风险边界。
- 启动、构建、测试或调试过程中实际运行过的命令和结果。
- 已验证依赖、未验证依赖和需要人工确认的依赖。
- 仍需咨询项目成员的问题，以及为什么这些问题会影响开发。

Agent 应把这些内容写入轻量项目熟悉报告，或明确建议用户保存到任务文档、团队文档或个人工作记录。报告不应成为项目百科，只记录对当前开发有价值的事实、证据和待确认事项。

### 2. 调试方法探索

Agent 不负责修改业务代码，但应帮助用户知道开发后如何顺利调试。

Agent 应识别并输出：

- 本地调试入口，例如 dev server、CLI、测试入口、服务入口或页面路由。
- 观察信号，例如日志位置、终端输出、浏览器控制台、网络请求、测试失败信息或健康检查。
- 最小复现路径，例如最小命令、最小页面、最小 API 请求或最小测试用例。
- 常见错误边界，例如环境变量缺失、依赖服务未启动、数据库未迁移、外部凭据缺失、缓存或队列未启动、端口冲突。
- 下一步调试动作，并且每次只建议一个低风险动作。

如果启动或验证失败，Agent 应帮助用户把问题拆成“配置问题、环境问题、真实依赖缺失、外部服务问题、代码问题或未知问题”。如果无法判断，应标记为 `[未知]` 或 `[需要确认]`，不能伪装成确定结论。

### 3. 真实依赖识别

Agent 必须区分不同类型的依赖来源：

- `[已确认]` 配置或 lockfile 中声明、代码中 import/use、脚本中实际调用，或命令输出已验证的依赖。
- `[基于证据]` README、docs、Docker 配置、CI 或错误输出共同指向，但尚未完整运行验证的依赖。
- `[需要确认]` 内部服务、私有凭据、团队专有环境、未公开数据库或口头约定依赖。
- `[未知]` 当前仓库和用户上下文无法判断的依赖。

Agent 不能因为 README 提到某个服务、目录名暗示某个框架、示例文件出现某个 SDK，就直接认定项目真实依赖它。依赖结论必须附带证据来源，例如文件路径、脚本名、配置项、代码引用或命令输出。

### 4. 防止表象误导

Agent 应显式检查可能误导判断的信号：

- README 与实际脚本不一致。
- 目录名、文件名或示例代码暗示某技术栈，但配置文件或入口代码不支持。
- 废弃文件、历史残留、未引用代码或测试 fixture 被误当成当前运行路径。
- lockfile、package manager、运行时版本或 CI 配置之间存在冲突。
- 命令没有实际运行成功，却被描述为“可运行”。
- 报错表面指向某模块，但根因可能是缺配置、缺服务或错误启动方式。

遇到证据冲突时，Agent 应把冲突本身作为重要结论记录下来，并说明当前更可信的证据是什么，以及还需要验证什么。

## 输出设计

### Agent 输出

现有输出结构应保留，但增加三个稳定区块或等价信息：

- `## 讨论中已确定并建议沉淀`
- `## 调试路径`
- `## 依赖与证据校验`

这些区块应简洁，不要求每轮都完整展开。只有当相关信息出现时才输出。

### 报告模板

报告模板应增加：

- 讨论结论记录。
- 调试路径。
- 真实依赖与证据。
- 证据冲突和误导风险。

## 验收标准

- `agent.md` 明确要求把与用户讨论确定的重要信息沉淀到报告中。
- `agent.md` 明确要求帮助用户探索开发后的调试方法，但不修改业务代码。
- `agent.md` 对真实依赖、推断依赖、内部依赖和未知依赖有清晰证据规则。
- `agent.md` 明确要求检查 README、目录名、示例文件、历史残留和报错表象带来的误导风险。
- README 的 Usage、Outputs、Human Confirmation 或 Limitations 同步体现这些新能力。
- 报告模板能记录讨论结论、调试路径、依赖证据和证据冲突。
- Agent 边界保持不变：仍然是编码前项目熟悉 Agent，不是 Coding Agent。
