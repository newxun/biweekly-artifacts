# Development Readiness PR #7 评审与优化设计

## 背景与结论

本设计基于两次 Claude Code 真实运行产物及
[`newxun/biweekly-artifacts#7`](https://github.com/newxun/biweekly-artifacts/pull/7)
的变更进行评审。Skill 的目标不变：帮助开发者在不熟悉的项目中，在编码前达到真实、
可验证、可交接的开发就绪状态。

PR #7 的方向基本正确：环境自举减少确认、增加 UI 健康检查、按任务拆分报告、识别任务
基准分支，并允许环境验证交给子 Agent。但当前实现仍存在授权边界过宽、健康检查不可操作、
基准分支概念混用、报告版本规则冲突、模板偏重及主/子 Agent 协作协议不完整等问题，
不建议按现状直接合并。

## 目标与非目标

### 目标

- 调用 Skill 后，无需为常规本地开发自举逐步确认。
- 对 UI 项目实际验证任务相关页面，而非仅验证端口或 HTTP 200。
- 支持登录受限场景，准确表达“有限就绪”。
- 每个任务独立确定基准分支；无线索时才回退远端默认分支。
- 同一会话持续维护同一组报告，新会话生成可比较的新版本。
- 将安装、构建、启动和运行验证的高噪音过程隔离到一个专项子 Agent。

### 非目标

- 不假设项目是微前端；仅在仓库证据表明存在宿主/子应用关系时采用对应分支流程。
- 不自动执行所有 build、test、lint；只执行由项目证据支持的最小验证矩阵。
- 不读取、输出或要求用户在对话中粘贴密码、Cookie、Token 等秘密值。
- 不因调用 Skill 而授权修改源码、配置、锁文件、Git 状态或共享数据。

## 一、授权模型

PR #7 中“环境自举默认安全”的表述不成立。安装脚本、测试和页面访问均可能产生外部或
持久副作用。应改为：

> 调用本 Skill 视为用户授予边界明确的本地开发自举权限。这些动作并非天然安全，必须
> 通过执行前预检、执行后核对和越界停止条件控制风险。

### 默认授权范围

- 检查工作树、运行时、包管理器、端口和已有进程。
- 执行从项目脚本、锁文件、CI 或项目文档交叉取证得到的安装、setup、开发启动命令。
- 生成项目已忽略的依赖、缓存、编译和临时产物。
- 执行必要且已确认无共享数据写入风险的构建、测试或 lint。
- 打开运行中的 UI、采集页面错误与截图。
- 停止并核对本流程启动的进程。
- 在 `docs/development-readiness/**` 创建或更新本会话报告。

### 必须停止并升级确认

- 修改被跟踪的源码、配置、锁文件或 Skill 报告之外的文档。
- 切换分支、创建 worktree、提交等 Git 写操作。
- 部署、迁移数据库、写入共享服务或运行可能修改真实数据的集成/E2E 测试。
- 连接生产环境、覆盖已有端口、停止非本流程创建的进程。
- 启动具有持久卷或共享数据风险的容器。
- 生命周期脚本存在未解释的用户目录写入、凭据读取或项目未声明的外连。

执行前后必须比较工作树和进程状态。出现意外跟踪文件修改时保留现场、停止并报告，
不得自行回滚或覆盖。

## 二、主 Agent 与环境子 Agent

支持子 Agent、能力匹配且环境执行预计耗时/高噪音或隔离确有收益时，应将环境执行委派给
一个“环境验证子 Agent”。简单低噪音项目、不支持子 Agent、能力不足或需要主 Agent 与
用户交互时，由主 Agent 执行或接管对应步骤。

### 主 Agent 职责

- 根据仓库证据确定验证矩阵、候选命令、目标页面和安全边界。
- 在派发前确认认证方式。
- 提供固定派发输入：cwd/Commit/工作树、验证矩阵、安全边界、目标环境、URL/渲染信号、认证策略、超时与 artifact 临时目录。
- 处理越界授权、必要的用户交互和基准分支歧义。
- 核验关键证据并对最终就绪结论负责。
- 将结构化结果写入报告，不把完整构建日志复制到主上下文。

### 认证方式确认

检测到 UI 需要登录时，主 Agent 只询问一次认证策略：

1. 复用已有浏览器登录态。
2. 打开可交互浏览器后由用户手动登录。
3. 使用已配置的测试凭据；用户只说明安全来源或变量名，不提供秘密值。
4. 当前无法取得登录条件。

不单独执行重复的浏览器登录预检。认证策略随环境验证任务一起交给子 Agent。平台不支持
子 Agent 与用户交互时，仅由主 Agent 接管手动登录步骤。
登录态无法安全共享时不假定可移交：由主 Agent 完成页面验证，或在平台支持时完成登录后
恢复同一子 Agent。

### 环境子 Agent 职责

- 在给定边界内完成安装、setup、启动及必要的构建/测试。
- 按指定认证方式打开应用和任务页面并采集运行证据。
- 调查环境失败，可应用不修改跟踪文件和共享状态的临时解决方案。
- 需要越界修复时停止，只返回证据和建议，不自行请求或扩大授权。
- 完成后清理自己启动的进程及临时认证状态。

### 固定返回契约

子 Agent 只返回摘要，完整日志保存在临时文件：

1. 执行基线：cwd、分支、Commit、执行前工作树。
2. 验证矩阵：检查项、命令、退出码、结果、证据范围。
3. 页面健康：URL、渲染信号、Console/Page Error、失败请求、截图路径、登录后页面状态。
4. 问题调查：原始现象、错误摘要、分类、尝试及结果、根因可信度。
5. 解决方案：已采用的临时方案、建议的持久修复、所需授权。
6. 副作用与清理：前后工作树、生成物、端口、进程及清理结果。
7. 推荐结论：就绪、有限就绪或未就绪，以及阻断和下一步。

认证无法继续时返回 `AUTH_REQUIRED` 和具体失败位置，不重复尝试或自行寻找凭据。主 Agent
至少复核工作树、进程清理、页面证据和阻断结论。

## 三、应用与任务页面健康检查

健康检查适用于所有有 UI 的项目；微前端只是检测到相应架构后的条件分支。

### 验证等级

- L1：开发服务启动、入口可访问。
- L2：应用公共页面或外壳正常渲染。
- L3：登录后任务相关页面正常渲染。
- L4：任务相关最小业务入口可触发，且不会修改共享数据。

任务就绪应优先达到 L3。L4 仅在项目证据支持且副作用安全时执行。页面验证至少记录：

- 从路由或项目配置取证得到的访问 URL 和最终 URL。
- 能证明页面完成渲染的关键 DOM 或等价信号。
- `pageerror`、Console Error、失败请求和关键接口状态。
- 截图或等价证据。
- 错误是否阻断当前任务，而不是简单按错误数量判定。

若存在宿主/子应用关系，分别记录独立启动和宿主集成是否验证；不能把微前端规则套给普通
项目。缺少宿主、登录态或测试账号时，不得声称任务页面已经验证。

## 四、就绪状态

环境和任务均使用三态：

- `就绪`：本范围内必要验证已通过。
- `有限就绪`：可以开始部分开发，但存在明确限制；报告必须列出允许范围、禁止范围和残余风险。
- `未就绪`：存在会阻断实现或必要验证的问题。

典型判断：

- 环境可启动但无法取得登录条件：环境可就绪，任务为有限就绪。
- 登录后任务页面白屏或发生阻断性运行时报错：任务未就绪。
- 无浏览器能力：页面结论降级为未验证，不得声称完全就绪。

状态由证据决定，不因用户是否接受而改变；但用户明确接受残余风险前，不得指示从“有限就绪”进入开发。

## 五、任务基准分支

基线报告只记录当前检出分支、环境验证分支及对应 Commit，不设置“后续任务默认基准分支”。
每份任务报告独立确定该任务的基准分支。

证据优先级：

1. 用户或任务系统明确指定。
2. 关联 PR 的目标分支、父需求说明。
3. 仓库团队规范。
4. Git ancestry、merge-base 和相关分支内容。
5. 分支名称仅作线索，不能单独确认。

用户、任务资料和仓库均没有任何相反线索时，回退到 `origin/HEAD` 指向的远端默认分支，
而不是机械优先 `main` 或 `master`。只读解析本地远端 HEAD，必要时用不更新本地引用的
`git ls-remote --symref origin HEAD` 交叉确认；无法可靠解析时询问，不猜分支或擅自 fetch。
发现父需求或特性分支线索时必须重新判断。

环境验证分支与任务基准分支不同时，按差异局部重验：

- 锁文件、运行时或构建配置变化：重跑安装、setup 和启动。
- 路由或应用入口变化：重做页面健康检查。
- 仅无关业务文件变化：复用环境结论并记录比较依据。

需要切换分支且当前工作区不适合直接操作时，应按授权协议申请一次 Git 操作，并优先使用
隔离 worktree。

## 六、报告路径与结构

报告采用“会话级版本”：会话启动时生成一个兼容 Windows 的 UTC 时间戳
`YYYY-MM-DDTHH-mm-ssZ`。同一会话从开始到结束持续更新同一组文件；创建和更新均无需
询问。新会话使用新时间戳创建新版本，旧版本保留用于 diff。

```text
docs/development-readiness/
├── 2026-07-17T08-38-00Z-baseline.md
└── tasks/
    ├── 2026-07-17T08-38-00Z-HWTQ-1652.md
    └── 2026-07-17T08-38-00Z-HWTQ-1655.md
```

一个任务一份文件，多个任务共享本会话基线。任务文件引用具体基线文件。只有新会话才创建
新版本，不因阶段推进、补充证据或同会话修订而新建文件。

### 基线文档核心结构

1. 当前结论与阻断。
2. 项目开发必需信息。
3. 环境启动与页面验证。
4. 证据及失效条件。

### 任务文档核心结构

1. 任务结论、限制和下一步。
2. 目标、边界与验收。
3. 修改面。
4. 开发及验证路径。
5. 阻断、风险与交接。

核心标题和信息位置固定；无内容的可选小节可以省略，不强制输出空表格或大量“不适用”。
同一事实只保留一个权威位置，其他位置使用引用。

## 七、端到端流程

1. 初始化会话时间戳和本会话报告路径。
2. 记录仓库与工作树基线，识别项目及最小验证矩阵。
3. UI 需要登录时，只确认一次认证方式。
4. 在平台支持时派发一个环境验证子 Agent。
5. 子 Agent 完成安全预检、环境自举、页面验证、问题调查和清理，返回固定摘要。
6. 主 Agent 核验关键证据，更新本会话基线报告。
7. 针对每个任务独立解析基准分支、定位修改面并按差异局部重验。
8. 尽可能达到登录后任务页面 L3 验证，无法达到时准确降级。
9. 每个任务更新本会话对应任务文件，完成跨会话交接。

## 八、PR #7 必须调整的内容

1. 将“环境自举默认安全”改成“Skill 调用授予有限自举权限”，补充预检和停止条件。
2. 用可执行的页面取证协议替代泛化的“打开页面无报错”。
3. 为任务增加“有限就绪”状态及登录降级规则。
4. 删除基线模板中的默认任务基准分支；增加 `origin/HEAD` 兜底规则。
5. 改为会话级时间戳文件，并取消首次创建/同会话更新询问。
6. 精简模板，固定核心结构但允许省略无内容的可选小节。
7. 统一 README、protocol 与评估文档中的子 Agent 触发规则和固定返回契约。
8. 直接修订原设计正文中的旧授权和报告规则，不能只追加相互矛盾的修订记录。

## 九、回归评估

除 PR 已有场景外，必须覆盖：

- Fresh clone 中常规安装、启动、页面验证和报告更新不重复询问。
- 生命周期脚本存在可疑副作用时停止并升级确认。
- 自举意外修改跟踪文件时保留现场并停止。
- 调度子 Agent 前只询问一次认证方式，不索取秘密值。
- 子 Agent 返回固定摘要，完整构建日志不进入主上下文。
- 无登录条件时任务有限就绪；登录后任务页面白屏时任务未就绪。
- 非 UI 项目不强制浏览器检查。
- 无分支线索时使用 `origin/HEAD`；父需求分支线索覆盖默认分支。
- 同一会话更新同一时间戳文件，新会话生成新版本。
- 多任务分别落盘并共享本会话基线。
- 无浏览器或子 Agent 能力时自然降级，不伪造验证结果。

两份真实 Claude Code 产物应作为人工回归样本，验证新结构能保留关键证据，同时明显减少
重复章节、空类目和执行日志噪音。
