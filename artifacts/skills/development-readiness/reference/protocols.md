# Protocols: 确认协议、专项子 Agent 与交接文档

> 由 `SKILL.md` 引用。在**要产生副作用**、**考虑派子 Agent**、或**生成/更新交接文档**时读取。

## 确认协议（副作用前置）

按风险分两档；调用 Skill 已授权第一档，不重复确认：

- **有限自举（已授权）**：只读取证；执行项目证据支持的安装、setup、启动，以及经脚本、配置和目标环境确认不会写本地/共享数据的必要验证；生成 ignored/临时产物；安全访问声明的开发/测试服务；进行 UI 页面取证；清理本流程进程；创建和更新本会话 `docs/development-readiness/**` 报告。执行前检查脚本、目标环境、端口和预期产物，执行后核对工作树与进程。它不是“天然安全”；副作用未知即越界。
- **越界副作用（先确认）**：修改报告外的跟踪文件或 Git 状态；写用户目录、本地/共享数据；启动有持久状态风险的容器；连接生产；部署、迁移、运行可能写真实数据的测试；项目未解释的外连；覆盖已有端口或进程。使用下述模板集中确认。

```markdown
## 修复批次确认

- 现象与证据：
- 根因判断及证据等级：
- 不修改的影响：
- 拟修改文件：
- 拟修改内容：
- 拟执行命令：
- 拟修改的非文件状态或外部目标：
- 网络访问的目标与传输内容：
- 是否保留进程、端口或容器：
- 影响范围：
- 项目修复还是临时绕过：
- 清理、恢复和回滚方式：
- 修改后的验证方式：

是否允许执行这个修复批次？
```

获批后：需要改跟踪文件时使用隔离分支或 worktree（无法隔离则停止）；保留用户原有改动；只改确认范围；展示 diff 并验证；报告原因与结果。一次授权不扩展到后续未说明修改。出现意外修改时保留现场并停止，不重置、不清理、不覆盖。

## 环境验证子 Agent

平台支持、子 Agent 具备 Shell、所需浏览器能力和进程清理能力，且执行预计耗时/高噪音或隔离能明显降低主上下文污染时，优先用**一个**环境验证子 Agent。简单、低噪音项目由主 Agent 直接完成。主 Agent 先从仓库证据确定验证矩阵、安全边界、目标页面；UI 需登录时先按 `methodology.md` 只确认一次认证方式，再派发。不要让主/子 Agent 重复执行同一套调查。

子 Agent 在一个任务内完成：预检 → 安装/setup → 启动 → 必要 build/test → 页面取证 → 问题调查 → 清理。它继承有限自举、秘密保护与网络边界，不得再次委派；需要改跟踪文件、Git 或共享状态时停止并返回建议，不能自行询问或扩大授权。认证不能继续时返回 `AUTH_REQUIRED` 与失败位置。

### 固定派发输入

主 Agent 提供：

1. cwd、当前分支/Commit、执行前工作树和禁止触碰的用户改动。
2. 验证矩阵、候选命令、允许/禁止副作用、目标环境、端口和超时。
3. UI 类型、目标 URL、期望验证等级和渲染信号。
4. 认证策略；若平台支持，提供不暴露秘密的共享浏览器会话句柄、Profile 或临时状态引用。
5. 日志、截图等 artifact 的临时目录与清理要求。

不能安全共享登录态时，不假定可移交：子 Agent 完成启动后返回认证点，由主 Agent 接管页面验证；或平台支持恢复同一子 Agent 时，主 Agent 完成手动登录后再恢复。认证状态文件只能放权限受限的临时目录，不写仓库、不输出内容，结束后删除。

### 固定返回契约

完整日志留在临时文件，返回摘要及 artifact 路径和有效期：

1. **执行基线**：cwd、分支、Commit、执行前工作树。
2. **验证矩阵**：检查项、命令、退出码、结果、证据适用范围、脱敏日志路径及有效期。
3. **页面健康**：URL、渲染信号、Console/Page Error、失败请求、截图路径及有效期、登录后页面状态。
4. **问题调查**：原始现象、错误摘要、分类、尝试及结果、根因可信度。
5. **解决方案**：已用临时方案、建议持久修复、所需授权。
6. **副作用与清理**：前后工作树、生成物、端口、进程和清理结果。
7. **推荐结论**：就绪/有限就绪/未就绪、阻断与下一步。

主 Agent 至少复核工作树、进程清理、页面证据和阻断，对最终状态负责。没有子 Agent、浏览器或交互能力时由主 Agent 串行完成可验证部分并降低受影响证据等级，不视为缺陷。

本 Skill 的项目熟悉、任务边界和用户决策仍由主 Agent 管理，不能整体塞进非交互子 Agent。复杂历史或依赖调查仍可委派一个边界明确、无需授权的专项子 Agent，使用同样的裁剪输入与核验原则。

## 交接文档与会话交接

交接文档必须自包含。会话启动时用 UTC 生成兼容 Windows 的时间戳 `YYYY-MM-DDTHH-mm-ssZ`（冒号改为连字符），本会话所有报告复用该时间戳：

- 基线：`docs/development-readiness/<timestamp>-baseline.md`
- 任务：`docs/development-readiness/tasks/<timestamp>-<task-id-or-slug>.md`

同一会话从开始到结束持续创建/更新这组文件，无需询问，不因阶段推进或修订新建版本。多个任务各一份并回链同一基线，不合并。新会话使用新时间戳，保留旧文件用于 diff。任务无 ID 时使用简短、稳定、文件系统安全的 slug；同名冲突追加短序号。

基线用 `report-template-base.md`，任务用 `report-template-task.md`。固定模板的核心标题与信息位置；无内容的可选小节可以省略，不保留空表格或堆叠“不适用”。同一事实只写在一个权威位置，其他位置引用。

基线报告记录仓库、当前分支、环境验证分支、Commit、工作树、环境、创建和更新时间；任务报告记录创建/更新时间、任务基准分支/Commit、环境验证分支、工作树及基线报告路径，共享环境信息只引用基线，不重复抄写。新会话先比较基线：一致则复用结论；发生变化时用 diff 只重验受影响部分并降低过期证据等级；无法判断则说明需重验内容。

输出时机：

- Phase 2 结束 → 创建或更新本会话基线。
- 每个任务 Phase 3 结束 → 更新基线并创建或更新该任务文件。
- 用户即将结束或切换会话 → 将当前检查点写入同一组会话文件。

## 最终就绪输出

完成 Phase 3 后输出：

```markdown
## 开发交接已准备

- 环境状态：就绪 / 有限就绪 / 未就绪
- 任务状态：就绪 / 有限就绪 / 未就绪 / 未评估（尚无任务）
- 交接完整度：最终交接 / 检查点
- 当前任务：
- 推荐修改范围：
- 环境验证：
- 开发与调试方式：
- 最大风险：
- 未解决阻断：
- 交接文档：

下一步：
- 任务状态为 `就绪`：在新的 Coding Agent 会话中读取交接文档并开始实现。
- 任务状态为 `有限就绪`：记录允许范围、禁止范围与残余风险；只有用户明确接受后才指示在允许范围内开始。
- 任务状态为 `未就绪`：先解除列出的阻断，不得开始实现。
- 任务状态为 `未评估`：补充具体任务后继续 Phase 3。

任何情况下都不要由本 Agent 继续编码。
```

若环境或任务状态不是 `就绪`，不得用「准备完成」掩盖阻断：标题改为「开发交接检查点」，明确用户必须先处理什么；任一状态为 `有限就绪` 时记录允许范围、禁止范围、残余风险及用户是否接受。
